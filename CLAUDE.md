## Project

University platform for "Certificate", a postgraduate institution (maestr√≠as/diplomados) in Bolivia. UI copy is in Spanish. pnpm workspaces monorepo: `frontend/` (Next.js 16 + React 19 + Tailwind v4 + NextAuth v5 + shadcn/ui + `socket.io-client`) and `backend/` (NestJS 11 + Prisma 7 + Zod + `@nestjs/websockets` over socket.io). Infra via `docker-compose.yml`: PostgreSQL 18, Redis, Garage (S3-compatible). **Real-time** (chat + notification push) runs over WebSocket gateways ‚Äî the browser connects directly to the backend's socket.io namespaces (`/chat`, `/notifications`) using the session's `accessToken`.


Env setup: copy `backend/.env.example` ‚Üí `backend/.env` (set `JWT_SECRET`) and `frontend/.env.example` ‚Üí `frontend/.env.local` (set `AUTH_SECRET`). Swagger at `http://localhost:4000/docs`. **Email is optional**: set `SMTP_*` (any provider ‚Äî Brevo/Resend/Gmail via SMTP) to send credential emails on user registration; left blank, the app runs and just logs that it skipped sending.

`pnpm dev` runs the backend's `dev` script, which is `concurrently` over `nest start --watch` **and** `prisma studio -p 51212` (visual DB browser at `http://localhost:51212`). `pnpm dev:backend` (= `start:dev`) runs only Nest, no Studio.

pnpm 10 blocks postinstall scripts by default; allowed packages live in root `package.json` `pnpm.onlyBuiltDependencies` ‚Äî extend it if a new native dep (e.g. bcrypt) fails to build.

## Production data safety (mandatory)

Every production deployment must preserve all existing database and file data. Never run seeds, resets, destructive migrations, `db push --accept-data-loss`, volume recreation, or any command that can delete or overwrite production data unless the user gives explicit, specific authorization for that destructive operation. Permission to deploy code alone is not permission to alter or delete data. If a schema change is not demonstrably additive, stop before deployment and request authorization with a concrete backup and impact plan.

## Architecture

### Auth flow (the core cross-app contract)

The **backend is the source of truth for identity and RBAC**. NextAuth does not manage users:

1. Frontend NextAuth v5 Credentials provider (`frontend/src/auth.ts`) POSTs to backend `/auth/login`.
2. Backend (`backend/src/auth/`) validates with Prisma + argon2 and signs a JWT (`sub`, `email`, `role`).
3. That JWT is stored in the NextAuth session as `session.accessToken` (jwt/session callbacks in `frontend/src/auth.config.ts`; types augmented in `frontend/src/types/next-auth.d.ts`). API calls send it as `Authorization: Bearer`.
4. Backend guards: `JwtAuthGuard` + `RolesGuard` with `@Roles(...)` decorator (`backend/src/common/`). Roles are the Prisma enum `Role`: `ADMIN | PROFESSOR | STUDENT`.
5. **"Recordarme" (login)** controls session lifetime. Checked ‚Üí backend signs a 30-day token (`JWT_REMEMBER_EXPIRES_IN`, default `30d`) and the NextAuth session cookie stays persistent; unchecked ‚Üí 1-day token (`JWT_EXPIRES_IN`) **and** the login server action downgrades the session cookie to a _session cookie_ (no `Max-Age`, cleared on browser close). The `remember` flag flows `login-form.tsx` ‚Üí `auth.ts` ‚Üí `POST /auth/login`; the cookie downgrade lives in `frontend/src/app/login/actions.ts` (`signIn(..., { redirect: false })` then re-set the `authjs.session-token` cookie without expiry). `session.maxAge` (`auth.config.ts`) is the 30-day upper bound; the real expiry is gated by decoding the backend token's `exp` (see `frontend/AGENTS.md`).
6. **Suspended account.** If credentials are correct but `User.isActive` is `false`, `POST /auth/login` returns **403** (not 401). The NextAuth credentials provider (`auth.ts`) throws a `CredentialsSignin` subclass with `code: "account_suspended"`; the login server action detects that code and shows "Tu cuenta ha sido suspendida. Comun√≠cate con administraci√≥n‚Ä¶". Passwords require **min 6 chars** across all DTOs/schemas (`login`/`create`/`update` + frontend Zod) ‚Äî bulk student rows have **no password field at all** (see the Users bullet below: it's auto-generated from the document).
7. **No public registration.** `POST /auth/register` was removed on purpose (anyone could self-create a STUDENT account and get a token): accounts are created only by the ADMIN via `/admin/users`. **Emails are normalized to lowercase** in the Zod DTOs (login + user create/update via `.transform`; bulk already lowercased) ‚Äî Postgres lookups are case-sensitive, so a mixed-case stored email would be unreachable at login.

### Prisma 7 specifics (backend)

- The client **requires a driver adapter**: `PrismaService` (`backend/src/prisma/prisma.service.ts`) passes `new PrismaPg(...)` to the constructor. `new PrismaClient()` without options throws.
- Datasource URL lives in `backend/prisma.config.ts`, not in `schema.prisma` ‚Äî the `datasource db` block must have **only** `provider` (no `url = env(...)`), or `prisma migrate dev` fails with `P1012: The datasource property url is no longer supported in schema files`.
- After editing `schema.prisma`, run `pnpm --filter backend exec prisma migrate dev --name <x>` then **restart the backend** ‚Äî the running `nest start --watch` process keeps the previously loaded `@prisma/client` in memory, so a regenerated client (new models/fields) isn't picked up until restart (`TS2339: Property 'x' does not exist on type 'PrismaService'`).
- **Migrations have drifted from the DB** (recent additions ‚Äî chat, `Program.displayOrder`, `Course.icon`, `CalendarReminder`, `ContentKind.FOLDER`/`FolderFile`, and the **activity-type** work: `SubmissionFile`, `ForumPost`, the quiz engine `Question`/`QuestionOption`/`QuizAttempt`/`QuizAnswer` + the `ModuleContent` quiz columns, the project **submission history** (`ProjectDelivery` + re-parenting `SubmissionFile` from `Submission` to `ProjectDelivery`, migrated with a hand-written SQL script that preserved existing files as "Entrega #1"), and the **landing `Program` flexibilization** (most fields now nullable ‚Äî only `title`/`categoryId`/`flyerUrl` required ‚Äî plus `specificObjectives String[]`, `hourlyLoad`, `extraFeatures Json` `[{label,value}]`, `ProgramTeacher.bio` + nullable `credentials`, `ProgramModule.order` min 0 for "M√≥dulo 0", and the payment fields: `installmentCount`/`installmentAmount`/`installmentCurrency` ("Plan de cuotas"; `totalCost`+`currency` = "Pago al contado"), `bankAccounts Json` `[{bank,accountNumber,holder?}]` (titular opcional/null; la landing omite su l√≠nea cuando falta), `qrImageUrl`, `videoUrl` (promo video: YouTube/Vimeo link or uploaded `/files/programs-videos/...`), the **`EnrollmentRequest` model + `EnrollmentRequestStatus` enum** for the landing's inscription form, the **`GalleryItem` model + `GalleryMediaType` enum** for the landing's gallery carousel, and the **`ClassSession` model** for the module class schedule) ‚Äî were applied with `prisma db push`, not migrations). `migrate dev` therefore wants to **reset** (data loss); for incremental schema changes use **`pnpm --filter backend exec prisma db push` + `prisma generate`** (no data loss) instead, then restart the backend. Reconciling the migration history into a fresh baseline is a separate, deliberate task.
- `prisma.config.ts` is excluded in `tsconfig.build.json` ‚Äî if it gets re-included, the build output nests under `dist/src/` and `node dist/main` breaks.

### Backend conventions

- Validation is Zod via `nestjs-zod`: DTOs are `class X extends createZodDto(schema)`; a global `ZodValidationPipe` is registered as `APP_PIPE` in `app.module.ts`. Do not introduce class-validator.
- `PrismaModule` and `QueueModule` (BullMQ over `REDIS_URL`) are `@Global()` ‚Äî inject `PrismaService` directly; register queues with `BullModule.registerQueue(...)`.
- A global `ThrottlerGuard` is active (100 req/min).
- **Public/admin module pattern** (`programs`, `partners`, `gallery`, `categories`, `settings`): each domain exposes a public controller with no guards (`findAll`/`findBySlug` ‚Äî only published/active rows, trimmed `select`) plus an `Admin*Controller` under `/admin/...` guarded by `JwtAuthGuard, RolesGuard` + `@Roles(Role.ADMIN)` (`findAllAdmin`/`findOneAdmin`/`create`/`update`/`remove`, full rows). Follow this split for new content-management domains.
- **Gallery** (`backend/src/gallery/`): the landing's photo/video gallery (`GalleryItem`, table `gallery_items`: `type` `GalleryMediaType` `IMAGE|VIDEO`, `url` relative `/files/...`, optional `title`, `displayOrder`, `isPublished`). Public `GET /gallery` (published only, ordered) + `/admin/gallery` CRUD with `PUT /admin/gallery/reorder` (same `orderedIds` helper as partners; create appends to the end). Files upload via `POST /uploads?folder=gallery` (images) and `POST /uploads/video?folder=gallery-videos` (the video endpoint takes an **optional `?folder=`** ‚Äî `programs-videos` default | `gallery-videos`). `update`/`remove` **best-effort delete the replaced/orphaned blob** (`StorageService.deleteByUrls`) after the DB write.
- **Site settings** (`backend/src/settings/`): singleton config row (`SiteSettings`, table `site_settings`, fixed id `"singleton"`) holding the landing's social links (`facebook`/`instagram`/`linkedin`/`youtube`/`tiktok`/`whatsapp`). `GET /settings` (public, social fields only); `GET /admin/settings` + `PATCH /admin/settings` (ADMIN). Service uses `upsert` by id `"singleton"` so the row self-creates; the Zod DTO normalizes empty string ‚Üí `null` and validates non-empty values as URLs. Extend this row (don't add new tables) for other global landing data.
- **Program categories** (`backend/src/categories/`): `ProgramCategory` is a managed table (`program_categories`), not an enum ‚Äî admins create/edit/reorder categories and assign each `Program.categoryId` to one. Slugs are auto-generated from `name` via `slugify` (`backend/src/common/utils/slugify.ts`) with a numeric-suffix retry loop for uniqueness. Deleting a category with `programs.count > 0` throws `ConflictException` (409). **Ordering is drag-and-drop, not a form field:** `categories`, `partners` **and** `programs` each expose `PUT /admin/<x>/reorder` (`{ orderedIds }` ‚Üí rewrites `displayOrder` 1..n); their create/edit forms no longer have an "orden" field (create appends to the end). The admin table order **is** the landing order (both read sorted by `displayOrder`). Same helper shape as `team`'s `reorder` (the original of the pattern).
- **Users / admin user management** (`backend/src/users/`): **admin-only** module (no public controller) following the `Admin*` pattern ‚Äî `/admin/users` (`JwtAuthGuard, RolesGuard` + `@Roles(Role.ADMIN)`) with `findAll(?role=)`/`findOne`/`create`/`update`/`remove` **+ `POST /admin/users/bulk`** (bulk student upload). Creates **PROFESSOR or STUDENT accounts only** (the create DTO's `role` enum excludes `ADMIN` ‚Üí `400`; duplicate email ‚Üí `409`); on **edit** the form **does not send `role`** (the partial update keeps the current role, incl. ADMIN). The `User` model carries **`phone` (required)**, **`idDocument` (required + `@unique`)**, **`gender`** (`Gender` enum `MALE|FEMALE`, **required**, `@default(MALE)` so `db push` backfills existing rows; the form renders a Base UI `Select`), and the **optional** `issuedIn` ("expedido en" ‚Äî place the document was issued, shown next to `idDocument`), `originUniversity`, and `profession`; passwords are argon2-hashed and **every response uses a `safeSelect` that omits the hash** (but includes all the above). Optional text fields normalize `""` ‚Üí `null` in the DTO. **`idDocument` is unique across all users**: a duplicate on create/update ‚Üí **409** (`uniqueConflictMessage` inspects the P2002 ‚Äî note Prisma 7's **driver adapter nests the constraint info** under `meta.driverAdapterError`, not `meta.target`, so the helper stringifies `meta` and looks for `idDocument`); bulk detects both in-file and DB duplicates (by email **and** document). Making the column required+unique was applied by first backfilling existing NULL/duplicate documents in the DB, then `db push --accept-data-loss`; the seed's admin now sets an `idDocument`. **Student passwords are auto-generated** (`generateStudentPassword` in `users.service.ts`): on a **new STUDENT** create ‚Äî single form **and** bulk ‚Äî the password is `firstName[0].toUpperCase() + lastName[0].toUpperCase() + idDocument` (e.g. Juan P√©rez / CI 1234567 ‚Üí `JP1234567`). The student form **hides the password field** and shows a live preview instead. **PROFESSOR** creation still uses a manual password (min 6, required by the service). **Editing never regenerates** the password (the edit form keeps the optional "leave blank to keep" field for any role). The frontend mirrors the generator in `user-schema.ts` (`generateStudentPassword`) purely for the preview ‚Äî keep the two in sync. **Names are stored UPPERCASE**: the create/update/bulk DTOs `.transform((v) => v.toUpperCase())` on `firstName`/`lastName` (existing rows were uppercased with `UPDATE ... UPPER()` in dev and prod on 2026-07-07; pre-change casing backups live in `.deploy/backup-nombres-*.csv`). **Name display convention ‚Äî "Apellido Nombre" (institutional order):** everywhere a full name is shown, sorted, searched, or built into a combined string, it is **`lastName` first, then `firstName`** (e.g. `` `${lastName} ${firstName}` ``, `orderBy: { lastName: 'asc' }`, initials `lastName[0]+firstName[0]`). This holds across backend (gradebook/quiz/forum `orderBy`, the chat/notification sender `name`) and frontend (every list/table/card/label, e.g. `UserForm`'s "Apellidos y nombre", search placeholders like "Buscar por apellido, nombre, correo o documento‚Ä¶"). `firstName` alone is still used for greetings. **Bulk upload** (`bulkCreateStudents`): body `{ students: [...] }` (rows parsed from an Excel **in the browser** with SheetJS/`xlsx`), validated **row-by-row** with Zod for a **partial load** ‚Äî creates valid rows as STUDENT, returns `{ total, created, errors[{ index, email, message }] }` (in-file + DB duplicate detection). The template columns are `Apellido`, `Nombre`, `Documento de identidad` (**required** ‚Äî it generates the password), `Expedido en`, `G√©nero`, `Tel√©fono`, `Correo electr√≥nico`, `Universidad de origen`, `Profesi√≥n` (**no password column**); the bulk `gender` cell is parsed leniently from Spanish text (`Femenino`/`F‚Ä¶` ‚Üí `FEMALE`, otherwise `MALE`) so a blank cell doesn't reject the row. This is the admin counterpart to `/auth/login`: the ADMIN creates the accounts that PROFESSOR/STUDENT later use.
- **Enrollment requests / Solicitudes de inscripci√≥n** (`backend/src/enrollment-requests/`): the landing's per-program inscription form ("Quiero inscribirme"/"Inscr√≠bete ahora" on `/programas/[slug]` ‚Üí `/programas/[slug]/inscripcion`). `POST /enrollment-requests` is **public** (throttled 5/min) and stores an `EnrollmentRequest` ‚Äî same fields as the student signup **minus password, all required** (same normalizations: names UPPERCASE, email lowercase) plus `programTitle`/`programSlug` snapshots. Admin side (`/admin/enrollment-requests`, ADMIN): list, `POST /:id/enroll` (converts to a STUDENT via `UsersService.create` ‚Äî auto-generated password + credentials email; marks `ENROLLED` + `enrolledUserId`; re-enroll or duplicate email/document ‚Üí 409), and DELETE. `UsersModule` **exports `UsersService`** for this. Frontend: dashboard section `/dashboard/solicitudes` ("Usuarios" nav group + home card with pending count) ‚Äî table with search, estado badge, per-row **WhatsApp** button and **"Inscribir"** (AlertDialog with the full data summary ‚Üí server action). The landing form is `enrollment-form.tsx` (RHF+Zod, dark landing styling) posting through the `submitEnrollmentAction` server action. **Each new request also notifies**: (a) every active ADMIN via a `ENROLLMENT_REQUEST` bell notification (live WebSocket push; action button "Ver solicitud" ‚Üí `/dashboard/solicitudes`), and (b) **an email** with the full applicant data + "Revisar e inscribir" / WhatsApp buttons, queued as an `enrollment-notice` job on the BullMQ `mail` queue (`MailService.enqueueEnrollmentNotice`/`buildEnrollmentNoticeEmail`; failures only log ‚Äî never notify-fail the public form). The recipient mailbox lives in `SiteSettings.enrollmentNotifyEmail` (default `certificatebolivia@gmail.com`, **not** exposed in the public `GET /settings`) and is edited inline from `/dashboard/solicitudes` (`NotifyEmailCard` ‚Üí `PATCH /admin/settings`).
- **Storage module** (`backend/src/storage/`): three upload endpoints + a public file server. `POST /uploads?folder=...` (ADMIN only, multipart, 5 MB, **image-mime** allowlist) for landing assets (flyers, logos, photos ‚Üí `flyerUrl`/`logoUrl`/`photoUrl`). `POST /uploads/video` (ADMIN only, multipart, **200 MB**, **video-mime** allowlist MP4/WebM/OGG/MOV: `uploadVideo`/`ALLOWED_VIDEO_MIME`) for a program's promo video ‚Üí folder `programs-videos` (frontend route handler `/api/admin/upload-video`, field `VideoUploadField`; the program form lets the admin either paste a YouTube/Vimeo link or upload a file, both stored in `Program.videoUrl`; the landing detail shows a "Ver video" button + modal `ProgramVideoModal` only when set ‚Äî iframe for links, `<video>` for uploaded files). `POST /me/uploads` (PROFESSOR, STUDENT **or** ADMIN ‚Äî the admin manages modules like a teacher, 20 MB, **document-mime** allowlist incl. images: `uploadDocument`/`ALLOWED_DOCUMENT_MIME`) for teacher/admin **materials** and student **submission** files ‚Äî the folder is chosen by role (STUDENT ‚Üí `submissions`, else `materials`), no query param. The forwarding route handler `/api/teacher/upload` must read `request.formData()` **before** any 4xx early-return: replying without consuming the multipart body leaves the browser's upload hanging forever (spinner never resolves). Both return a **relative** path `/files/<folder>/<uuid>.<ext>`; `GET /files/:folder/:filename` streams it back publicly. **Return relative, never absolute** (`http://localhost:4000/...`): the frontend proxies `/files/*` to the backend via `rewrites()` in `next.config.ts` so `next/image` treats uploads as local images ‚Äî Next 16's image optimizer rejects remote URLs that resolve to private IPs like `localhost` (`"url" parameter is not allowed` / `resolved to private ip`). **Blob deletion:** `StorageService.deleteByUrls(urls)` is a **best-effort** cleanup (`DeleteObjectCommand`) ‚Äî it maps each relative `/files/...` back to its S3 key (`keyFromUrl`; ignores external links/videos), dedupes, and never throws if an object is already gone (only logs). Callers run it **after** the DB delete commits so a failed cleanup leaves orphans (logged), never dangling DB refs. Cascade-deleting DB rows does **not** auto-remove blobs, so any code that deletes file-bearing rows must call this ‚Äî current callers: `ModuleContentService.removeContent`, `CoursesService.remove` + `update` (removed modules, via `collectModuleBlobUrls`), and `UsersService.remove` (a deleted STUDENT's submission files). A one-off sweep `backend/scripts/clean-orphan-blobs.ts` (`ts-node`, dry-run by default, `--delete` to act) removes orphans under `submissions/`/`materials/` not referenced by `ModuleContent.url`/`FolderFile.url`/`Submission.fileUrl`/`SubmissionFile.url` (leaves landing folders untouched).
- **Mail** (`backend/src/mail/`): transactional email over **SMTP with `nodemailer`** (provider-agnostic ‚Äî configure `SMTP_HOST/PORT/USER/PASS/FROM`; Brevo's free 300/day is the default recommendation, Resend/Gmail also work). `MailService` builds the transport in `onModuleInit` (disabled + logs a warning if `SMTP_*` is unset ‚Äî `isEnabled=false`, sends become no-ops so registration never fails on email). **Sending is queued, not inline**: `enqueueCredentials(payload)` adds a `credentials` job to the **BullMQ `mail` queue** (`MailProcessor`, `concurrency: 5`, 3 attempts + exponential backoff) so bulk student uploads don't block the request or hammer SMTP. `UsersService.create` (single) and `bulkCreateStudents` (per created row) enqueue a **welcome email with the plaintext credentials** (email + password ‚Äî captured at creation time, before hashing) and the login URL. That URL is `MAIL_LOGIN_URL` if set, else the first origin of `FRONTEND_URL` + `/login` (`MailService.loginUrl`) ‚Äî kept as a **separate env var** on purpose: `FRONTEND_URL` also drives CORS/WebSocket origins (`main.ts`, chat/notifications gateways) and can list several comma-separated origins (dev + prod), so the mail link shouldn't silently depend on which one sorts first. The Spanish HTML+text template is `buildCredentialsEmail` (exported, pure ‚Äî HTML-escapes user values). `MailModule` exports `MailService` and is imported by `UsersModule` (and imports `NotificationsModule` ‚Äî see next). **Failed sends notify the ADMIN**: when a credentials job exhausts its 3 attempts (SMTP quota/outage/immediate reject ‚Äî **not** async bounces, which land in the sender's mailbox), `MailProcessor`'s `@OnWorkerEvent('failed')` creates a **`MAIL_FAILED`** notification for every active ADMIN with the failure reason and the credentials in `data` (email/password/phone/loginUrl ‚Äî the payload carries `lastName`/`phone` for this); the frontend renders a green **"Enviar credenciales por WhatsApp"** action button on it (`notification-action.ts`, `kind: "whatsapp"`, opens `wa.me` prefilled). The handler never throws (a notify failure only logs).
- **Notifications** (`backend/src/notifications/`): one `Notification` row per recipient. `NotificationsService` (`@Module` exports it so other modules inject it) has `listForUser`/`unreadCount`/`openForUser` (returns + marks read)/`markRead`/`markAllRead` and a `createMany(inputs, tx?)` fan-out helper. `NotificationsController` (`/notifications`, `JwtAuthGuard` only ‚Äî each user operates only their own). `AdminNotificationsController` (`/admin/notifications`, ADMIN) sends broadcasts (`broadcast(dto, senderId)` resolves the audience to active PROFESSOR/STUDENT ids, logs an `Announcement`, and `createMany`s the notifications in a transaction) and lists the paginated/filterable history. Other services emit notifications by injecting `NotificationsService`: `CoursesService` (enrollment, module-teacher assignment), `GradingService` (graded), `ModuleContentService` (activity published), `ChatService` (chat `MESSAGE`).
  - **Real-time push (WebSocket, no polling).** `NotificationsGateway` (socket.io namespace `/notifications`, JWT auth in the handshake) joins each user to room `user:<id>`. `createMany` now uses **`createManyAndReturn`** and, when **not** inside a transaction, immediately pushes each row as `notification:new`; when inside a `tx` (broadcast/enrollment/teacher-assignment) it returns the rows and the caller emits **after commit** via `emitNotifications(rows)` (so a rolled-back tx never pushes a phantom). `emitConversationRead(userId, conversationKey)` pushes `notification:read` when a chat is opened so the bell clears those notifications live. The bell (`NotificationBell`) subscribes to both events; there is **no** `GET /api/notifications` polling anymore.
- **Chat** (`backend/src/chat/`): direct 1-to-1 messaging **teacher ‚Üî student inside a module** (replaces the removed "Apuntes"). `ChatService` authorizes by teaching and enrollment (the student only with the module's teachers, not on DRAFT modules; the teacher only with enrolled students). **Conversations require the teacher side to be a real `ModuleTeacher`** (`resolveConversation`): the ADMIN can read contacts but cannot open/send a conversation ‚Äî the student's contact list only shows module teachers, so an admin-side conversation would be invisible to the student (the UI already hides the Chat tab for ADMIN). REST (`ChatController`, `/me/chat`, `JwtAuthGuard`): `GET /me/chat/:moduleId/contacts` (counterparts + unread-per-contact) and `GET /me/chat/:moduleId/contacts/:counterpartId/messages` (history; marks incoming read + emits `notification:read`). **Sending goes over WebSocket** (`ChatGateway`, namespace `/chat`): the client emits `chat:send`; the gateway persists via `ChatService.sendMessage`, creates a collapsed `MESSAGE` notification for the recipient, and re-emits `chat:new` to `user:<sender>` and `user:<recipient>`. The gateway sets the socket identity **synchronously** after verifying the JWT (before any `await`) to avoid a race where a `chat:send` arrives before `client.data` is populated. A conversation is keyed by `(moduleId, studentId, teacherId)`; `senderId` gives direction. (`ChatMessage` model ‚Äî see DATAMODEL.md.)
- **Calendar** (`backend/src/calendar/`, `/me/calendar`, `JwtAuthGuard`): feeds both variants of `ProfileCalendar`: the compact dashboard right-panel calendar and the expanded PROFESSOR/STUDENT page at `/dashboard/calendario` (linked from the left sidebar). `GET /me/calendar/overview` returns `{ deadlines, reminders, classes }` ‚Äî `deadlines` are the `dueDate`s of `ACTIVITY` contents in the user's courses (student: enrolled/`ACTIVE|FINISHED`/non-DRAFT/published; teacher: taught modules; admin: none); `reminders` are the user's own notes; `classes` are the module **class-schedule sessions** with the same role visibility as deadlines (see Schedule module). New activities/grades default to **`weight` 100%**; new courses default to **`passingScore` 71** (form/DTO). **Admin "Notas de estudiantes" reads** (`backend/src/courses/admin-student-grades.controller.ts`, `@Roles(ADMIN)`): `GET /admin/students/:id/kardex` (reuses `getKardex`) and `GET /admin/students/:id/grades` (`getStudentGradeDetail` ‚Äî per-activity detail: programs ‚Üí modules ‚Üí each `ACTIVITY` score + the final `ModuleGrade` + observation). Same visibility rules as the student's own reads (ACTIVE/FINISHED courses, no DRAFT modules). **Each `ActivityType` now behaves differently** (it used to be just a label): `ASSIGNMENT` = text + **one** file (`Submission.fileUrl`); `PROJECT` = **submission history** ‚Äî each submit **creates a new `ProjectDelivery`** (its own `SubmissionFile[]`) instead of replacing; previous deliveries are **immutable** and the whole activity keeps **one** grade on the `Submission` (`submitActivity` upserts the `Submission` for status and appends a `ProjectDelivery` with `order=max+1`; reads return `deliveries` newest-first ‚Äî see DATAMODEL.md ‚Üí `ProjectDelivery`); `FORUM` = public discussion thread (see Forum module); `QUIZ`/`EXAM` = auto-graded question engine (see Quiz module). **Invariant preserved:** every type still ends up writing `Submission.score` (manual for Tarea/Proyecto/Foro participation; automatic for Quiz/Examen), so `recomputeModuleGrade` and the k√°rdex are untouched ‚Äî `GradingService` exposes `applyQuizResult(...)` (used by the quiz engine) and a now-public `recomputeModuleGrade(studentId, moduleId, gradedById|null)`.

- **Quiz engine** (`backend/src/quiz/`, `/me/quiz`, `JwtAuthGuard`): the `QUIZ`/`EXAM` question engine. Teacher: `GET /:activityId/editor` + `PUT /:activityId/questions` (replaces the whole question bank), `GET /:activityId/attempts`, `GET /attempts/:attemptId`, `PUT /attempts/:attemptId/grade-essays`. Student: `GET /:activityId` (runner state: availability window, attempt, questions **without** correct answers ‚Äî shuffled per `attemptId` if `shuffle`), `POST /:activityId/start` (validates window + single-attempt), `POST /:activityId/submit`. **Auto-grading** in `submit` scores `SINGLE_CHOICE`/`MULTIPLE_CHOICE` (exact set match by option id), `TRUE_FALSE`, and `SHORT_TEXT` (normalized accept-list); **`ESSAY`** answers stay ungraded ‚Üí the attempt is `SUBMITTED` until the teacher grades essays, then `GRADED`. Scores scale to `maxScore` and flow to `Submission`/`ModuleGrade` via `GradingService.applyQuizResult`. **Quiz settings** (`timeLimitMin`/`availableFrom`/`availableUntil`/`singleAttempt`/`shuffle`/`revealAnswers`) live as nullable columns on `ModuleContent`, set from the content-creation form. (`Question`/`QuestionOption`/`QuizAttempt`/`QuizAnswer` models ‚Äî see DATAMODEL.md.) `QuizModule` imports `GradingModule` (which now `exports` `GradingService`). **Integrity rules (server-side, the client timer is not authoritative):** the effective attempt deadline is `min(startedAt + timeLimitMin, availableUntil)` (`attemptDeadline`) with a **60s grace** (`SUBMIT_GRACE_MS`); a `submit` past it ‚Äî or a stale `IN_PROGRESS` attempt found by `getRunner` ‚Äî is **force-closed grading the autosaved answers** (`expireAttempt`; 0 if nothing was saved; if an essay was answered it goes `SUBMITTED` for teacher review; `submit` returns `expired: true` and the runner shows a "se calific√≥ con lo autoguardado" toast). **Progressive autosave:** `PUT /:activityId/answers` (`autosave`) upserts the in-progress attempt's `QuizAnswer`s **ungraded** (`isCorrect`/`pointsAwarded` null until submit) ‚Äî the runner debounces it 2.5s per change + a `keepalive` flush on `pagehide`, and `getRunner` returns `savedAnswers` so a page reload restores the form. This is why an expired/abandoned attempt keeps the student's work instead of zeroing it. **`singleAttempt = false` allows retries**: the runner response includes `canStart` on a `GRADED` attempt (still open, module not FINISHED) and the UI shows "Intentar de nuevo" ‚Äî the latest attempt's score replaces the previous one; retrying is blocked while essays are pending (`SUBMITTED`). **The question bank locks once attempts exist**: `saveQuestions` throws 409 (editing would cascade-delete the students' `QuizAnswer`s) and `getEditor` returns `attemptCount` so `QuizManager` disables the editor with a notice. `gradeEssays` rejects `IN_PROGRESS` attempts (grading one would close it with 0 mid-attempt). `start` blocks when the latest attempt is `SUBMITTED` and resolves the double-click race (P2002 ‚Üí resume the winner).
- **Recuperatorio / Segunda instancia** (`ModuleContent.recoveryStage`, enum `RecoveryStage`): a **recovery exam** enabled on a **FINISHED** module for failing students ‚Äî the one exception to the module-finished write-block. It's a normal QUIZ/EXAM-engine activity but it does **not** weight: if the student takes it (GRADED submission), the module `finalScore` becomes **the highest of {weighted module grade, recuperatorio, segunda instancia} with the recovery scores (base 100) capped at the course's `passingScore`** (the program's minimum passing grade, default 71 ‚Äî **not** a hardcoded 71) ‚Äî a final grade above the minimum is only possible without entering recovery, and a recovery score lower than the current grade never lowers it (`recomputeModuleGrade` excludes recovery contents from the weighted average and applies `max(weighted, min(recoveryScore, passing))`; `recoveryOverride` returns the max of the two stages; not taking any keeps the original grade). The rule is unit-tested in `grading.service.spec.ts` (mocked Prisma); grades stored under the old "recovery replaces" rule are fixed by the one-off `backend/scripts/recompute-recovery-grades.ts` (ts-node, **writes** ‚Äî recomputes only student√ómodule pairs with a graded recovery submission, logging before ‚Üí after). **`RECUPERATORIO`** is created by the teacher (or admin) from the module workspace ("Habilitar recuperatorio" button, shown when the module is FINISHED and `failedCount > 0`); **`SEGUNDA_INSTANCIA` only by the ADMIN** and only if a recuperatorio exists (service validates: FINISHED module, QUIZ/EXAM type, one per stage ‚Üí 409). Eligibility (`GradingService.recoveryEligibleStudents`): grade `FAILED` for recuperatorio; for segunda instancia additionally a GRADED recuperatorio submission. Non-eligible students never see the content (`getModuleForStudent` filters; `getRunner`/`start` 404); publishing notifies **only** eligible students. `QuizService` skips the FINISHED block for recovery contents (start/submit/autosave/saveQuestions/gradeEssays); the runner reports `moduleFinished: false` for them and includes `activity.recoveryStage` + `activity.passingScore` (the UI shows the "se toma la nota mayor, m√°ximo `passingScore`" banner + rose/purple badges in temario, gradebook, teacher activity page and content manager). Deleting the recovery content restores the weighted grade (recompute after cascade).

### Infra gotchas

- Garage's `command` needs the `--default-bucket` flag for the `GARAGE_DEFAULT_*` env vars to auto-create the bucket/key. S3 clients must use `endpoint: http://localhost:3900` with `forcePathStyle: true` (creds in `backend/.env` `S3_*`).
- Postgres auth method is baked into the volume at first init ‚Äî changing auth-related env vars in docker-compose requires recreating the `postgres_data` volume.
- **macOS port 5432 conflict:** if Postgres.app (or any local Postgres) is running, it binds `127.0.0.1:5432`/`[::1]:5432` ahead of Docker's wildcard bind, so the backend connects to the _wrong_ server (`P1010 / role "university_user" does not exist`) even though `psql` via Docker works. Stop the local one: `pg_ctl -D "~/Library/Application Support/Postgres/var-16" stop`. Disable its autostart to avoid recurrence.
