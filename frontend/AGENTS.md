## Next.js 16 gotchas encontrados en este repo

- **`next/image` bloquea IPs privadas (SSRF):** el optimizador rechaza URLs remotas que resuelven a `localhost`/`127.0.0.1`/`::1` con `400 "url" parameter is not allowed` (log: `resolved to private ip`). Por eso las imágenes subidas se guardan como **rutas relativas** `/files/...` y `next.config.ts` hace `rewrites()` de `/files/*` al backend, de modo que `next/image` las trata como imágenes locales. No uses `remotePatterns` apuntando a `localhost`. (Ver CLAUDE.md → Storage module.)
- **`useSearchParams()` exige `<Suspense>`:** un client component que lo use debe ir envuelto en `<Suspense>` o el build de producción falla (_Missing Suspense boundary_). En dev "funciona" sin él — no te fíes. Ejemplo: `programs-grid.tsx` envuelto en `programs.tsx`.
- **View Transitions (morph de elementos compartidos) entre card y detalle.** El componente `<ViewTransition>` **de React** (no una API de Next aparte) se habilita con `experimental.viewTransition: true` en `next.config.ts`. Para el morph: envolver el **mismo elemento** en origen y destino con `<ViewTransition name="...">` usando un `name` **único y estable** por entidad (aquí `program-image-${slug}` / `program-title-${slug}` en `programs-grid.tsx` y `programas/[slug]/page.tsx`). Funciona tanto en client como en server components. Gotchas encontrados:
  - **El morph solo se dispara en navegaciones que pasan por la integración de Next** (`<Link>`, `router.push`/`replace`). **`router.back()`/popstate NO lo dispara** (el morph "desaparece"). Para volver con morph, navega **hacia adelante** a la URL de origen, no con `back()`.
  - **`scroll-smooth` global compite con el morph:** estaba en `<html>` (`layout.tsx`) y al navegar animaba el reset de scroll como un "deslizamiento al tope" que tapaba la transición. Se quitó; el scroll suave de los anclajes del landing ya lo hace `ScrollLink` con `scrollIntoView({ behavior: "smooth" })` en JS, así que no depende de esa clase.
  - **Restaurar scroll para que el morph aterrice en el sitio real:** al volver, navegamos con `router.push(href, { scroll: false })` y restauramos la posición guardada en un **`useLayoutEffect`** (patrón isomórfico en `restore-landing-scroll.tsx`), **no `useEffect`**: el snapshot "nuevo" de la View Transition se captura tras los layout effects y antes del paint, así que el scroll debe fijarse en esa fase o el morph aterriza en la posición equivocada y la página pega un salto. La URL + `scrollY` se guardan en `sessionStorage` al hacer clic en la card (`programs-grid.tsx`), y una marca `landingRestore` evita restaurar en visitas normales o al ir a otra ancla como `#contacto`.
- **No `setState` en `useEffect` para sincronizar estado derivado de la URL/props:** la regla `react-hooks/set-state-in-effect` lo prohíbe. Ajusta el estado durante el render comparando con el valor previo (patrón recomendado por React); ver `programs-grid.tsx`.
- **`next.config.ts` lee env al evaluar:** usa `process.env.API_URL ?? process.env.NEXT_PUBLIC_API_URL ?? "http://localhost:4000"` para el destino del proxy.
- **React 19 resetea `<form action={fn}>` tras cada submit:** los inputs **no controlados** vuelven a su estado inicial al completar la action (incluso en fallo). Para conservar un valor tras un intento fallido (p. ej. el correo del login al equivocarse de contraseña) hazlo **controlado** y re-siémbralo desde el estado que devuelve la action — no confíes en `defaultValue`. Ejemplo: `src/app/login/login-form.tsx` (email controlado desde `state.values.email`, re-sincronizado durante el render comparando el valor previo, no en `useEffect`). La server action `authenticate` devuelve el correo **crudo** (antes de validar con Zod) en `LoginState.values.email` para re-hidratar incluso si es inválido.
- **Un Server Component NO puede escribir/borrar cookies en Next 16** — solo los **Route Handlers** y las **Server Actions** pueden. Consecuencia: si un Server Component (p. ej. una lectura admin durante el render) necesita cerrar sesión / limpiar la cookie, no puede llamar a `signOut` directamente; debe **redirigir a un Route Handler** que lo haga. Ver `src/app/api/auth/session-expired/route.ts` (`signOut({ redirect:false })` + `NextResponse.redirect`).
- **"Recordarme" = downgrade de la cookie de sesión en la Server Action.** NextAuth siempre escribe la `authjs.session-token` como cookie **persistente** (`Max-Age = session.maxAge`), marques o no la casilla — así que el flag por sí solo no cambia nada observable. Para que "sin Recordarme" cierre sesión al cerrar el navegador, `src/app/login/actions.ts` llama `signIn("credentials", { ..., redirect: false })` (para recuperar el control tras autenticar) y, si **no** se marcó, **reescribe** la cookie con el mismo valor pero **sin `Max-Age`/`Expires`** (cookie de sesión), vía `cookies()` de `next/headers` (permitido en Server Actions). Maneja la variante `__Secure-` y los fragmentos `.0/.1`. Con `redirect:false`, una credencial inválida **no lanza**: `signIn` devuelve una URL con `?error=` (se detecta para mostrar el error). El TTL real lo sigue gobernando el `exp` del token del backend (1 día vs 30 con `JWT_REMEMBER_EXPIRES_IN`), no el `maxAge`.
- **La sesión de NextAuth puede sobrevivir al `accessToken` del backend → bucle de redirección.** La sesión JWT de Auth.js tiene su propio `maxAge` y es _rolling_ (se renueva en cada request), mientras el `accessToken` del backend tiene TTL fijo (`JWT_EXPIRES_IN`). Si el token caduca/queda inválido (vencido, usuario `isActive=false`, DB reseedeada) pero la sesión sigue "viva", las llamadas admin reciben **401** aunque `auth()` devuelva sesión. Un `redirect("/login")` ingenuo entra en bucle: `login/page.tsx` ve la sesión válida y rebota a `/dashboard` → 401 → `/login` → … (`ERR_TOO_MANY_REDIRECTS`). Solución aplicada (ver `src/auth.config.ts`, `src/lib/api/admin.ts`): (1) `session.maxAge` es la **cota superior** = el token más largo posible (30 días, el de "Recordarme"); el `exp` real del `accessToken` es la fuente de verdad de la duración; (2) los callbacks `authorized` y `session` decodifican el `exp` del `accessToken` (`isBackendTokenExpired`, edge-safe con `atob`, **no** `Buffer`) y lo invalidan proactivamente; (3) los 401 del path admin redirigen a `/api/auth/session-expired` (Route Handler que cierra la sesión muerta y manda a `/login?expired=1`), rompiendo el bucle.
- **`next-themes` con modo oscuro por defecto + guard de hidratación.** El panel usa `next-themes` (`ThemeProvider` en el root `layout.tsx`: `attribute="class"`, `defaultTheme="dark"`, `enableSystem={false}`, `disableTransitionOnChange`). next-themes fija la clase del tema (`.dark`) **en el cliente antes de hidratar**, así que el `<html>` **debe** llevar `suppressHydrationWarning` (si no, warning de hidratación por el atributo `class`). Cualquier componente que lea el tema (`useTheme()`) tiene que **diferir el render del estado real hasta estar montado** para no divergir del SSR — lo resuelve `theme-toggle.tsx` con un `useMounted()` basado en **`useSyncExternalStore`** (no `useEffect`, por la regla `react-hooks/set-state-in-effect`), mostrando un placeholder del mismo tamaño mientras tanto. El mismo patrón `useSyncExternalStore` respalda el estado del shell persistido en `localStorage` (colapso del sidebar, panel de perfil): `getServerSnapshot` devuelve el default = HTML del SSR, evitando el mismatch.
- **El `Tooltip` de shadcn aquí es Base UI, no Radix.** `src/components/ui/tooltip.tsx` se basa en **`@base-ui/react`** (`Tooltip as TooltipPrimitive` de `@base-ui/react/tooltip`), no en `@radix-ui/*`: la API difiere (`TooltipTrigger render={...}`, `Positioner`/`Popup`/`Portal`, `delay` en el `Provider`). Base UI **portala el popup al `body`**, justo lo que necesita el riel minimizado del sidebar — su contenedor tiene `overflow-y-auto`, que recortaría un tooltip renderizado inline. Úsalo con `TooltipProvider` arriba (el sidebar lo monta una vez en `DashboardSidebar`).
- **`Popover` de Base UI (campanario de notificaciones).** `src/components/dashboard/notification-bell.tsx` usa `Popover` de `@base-ui/react/popover` (no Radix): `Popover.Root` (controlado con `open`/`onOpenChange` para cerrarlo al navegar), `Trigger`/`Portal`/`Positioner`/`Popup`. El estado abierto del trigger es el atributo `data-popup-open` (estíla con `data-[popup-open]:...`). Portala al `body`, así no lo recorta el `overflow` del topbar.
- **`Dialog` (modal) también es Base UI, no Radix.** `src/components/ui/dialog.tsx` se basa en `@base-ui/react/dialog` (espeja `alert-dialog.tsx`): `Dialog`/`DialogContent`/`DialogHeader`/`DialogTitle`/`DialogDescription`, backdrop con blur, cierre por X/Escape/backdrop, portala al `body`. Se usa para los flujos de configuración del docente (selector de tipo de contenido, crear/editar contenido, "Nueva calificación"/editar en la libreta). **Para contenido alto** (el editor Tiptap, formularios largos) aplica `max-h-[85vh] overflow-y-auto` en el `DialogContent` del consumidor para que el cuerpo haga scroll y no se corten los botones. **No** uses shadcn-add para Dialog (jala Radix).
- **Tiptap requiere `immediatelyRender: false` (SSR).** El editor de Temas (`src/components/dashboard/rich-text-editor.tsx`, `@tiptap/react` + StarterKit + Link) **debe** pasar `immediatelyRender: false` en `useEditor` o Next lanza mismatch de hidratación. El editor se monta/desmonta con su `Dialog`, así que se reinicializa por cada apertura (esperado, sin `useEffect`). Para **mostrar** el HTML resultante usa `RichTextContent` (`rich-text-content.tsx`) con la clase `.richtext` (estilos en `globals.css`).
- **Reordenar con `@dnd-kit` + estado local sincronizado sin `useEffect` ni `ref` en render.** `content-manager.tsx` mantiene el orden en estado local (drag-and-drop optimista) y lo sincroniza con el servidor comparando una **firma** durante el render guardando el valor previo **en estado** (`const [prevSig,setPrevSig]=useState(...); if (sig!==prevSig){setPrevSig(sig); setItems(...)}`) — **no en un `ref`**: la regla `react-hooks/refs` prohíbe leer/escribir `ref.current` durante el render. (Combina con la regla `set-state-in-effect`: ni effects ni refs para estado derivado.) El endpoint de reorden acepta una lista **parcial** (solo los visibles del temario; las actividades `isOffline` se ocultan de él y se mantienen al final).
- **Estado persistido con "auto-abierto que el usuario puede sobreescribir".** El árbol de Programas del sidebar (`useProgramOpen`) distingue _no seteado_ de _seteado_: lee el snapshot **crudo** del store (`"0"|"1"|null`) y computa `open = raw===null ? autoOpen : raw`, con el `toggle` persistiendo el **opuesto del valor visible**. Así el programa activo arranca abierto pero el botón lo puede cerrar de forma fiable (el viejo `open || active` rompía el toggle). `getServerSnapshot`/hidratación devuelven el default → sin mismatch.
- **En un archivo `"use server"` TODO export debe ser `async`.** Una server action exportada que solo `return`a una promesa sin declararse `async` rompe el **build de producción** (Turbopack: _"... had an error"_), aunque `tsc`/dev pasen. Declara `export async function ...`. Ej.: `dashboard/modulos/[moduleId]/actions.ts`.
- **`module` es nombre de variable prohibido.** La regla `@next/next/no-assign-module-variable` falla con `const module = ...`. Usa otro nombre (`mod`, `courseModule`). Aparece al traer un módulo del LMS en un page server component.
- **Acordeón sin JS con `<details>`/`<summary>`.** Las listas de módulos (`mis-cursos/[id]/course-module-item.tsx`) usan `<details open={...}>` nativo — server-rendered, cero cliente. Oculta el marcador con `[&::-webkit-details-marker]:hidden list-none` y rota el chevron con `group-open:rotate-180`.
- **Imprimir a PDF = ruta independiente fuera del layout del panel.** Toda ruta bajo `/dashboard/*` hereda el layout del panel (sidebar + topbar), que ensuciaría la impresión. La vista imprimible del kárdex vive en `src/app/kardex-pdf/` (ruta de nivel superior, solo root layout) con estilo claro **explícito** (no tokens de tema, para que el PDF sea consistente), `window.print()` desde un client component, barra oculta con `print:hidden`, `@page { margin }` en un `<style>`, y `print-color-adjust: exact` para que impriman los fondos de color. El logo blanco no sirve aquí (fondo blanco): usar wordmark de texto en navy.
- Recuerda: middleware es `src/proxy.ts` (no `middleware.ts`); ver CLAUDE.md para el contrato de auth.
