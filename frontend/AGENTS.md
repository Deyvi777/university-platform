## Next.js 16 gotchas encontrados en este repo

- **`next/image` bloquea IPs privadas (SSRF):** el optimizador rechaza URLs remotas que resuelven a `localhost`/`127.0.0.1`/`::1` con `400 "url" parameter is not allowed` (log: `resolved to private ip`). Por eso las imágenes subidas se guardan como **rutas relativas** `/files/...` y `next.config.ts` hace `rewrites()` de `/files/*` al backend, de modo que `next/image` las trata como imágenes locales. No uses `remotePatterns` apuntando a `localhost`. (Ver CLAUDE.md → Storage module.)
- **`useSearchParams()` exige `<Suspense>`:** un client component que lo use debe ir envuelto en `<Suspense>` o el build de producción falla (_Missing Suspense boundary_). En dev "funciona" sin él — no te fíes. Ejemplo: `programs-grid.tsx` envuelto en `programs.tsx`.
- **No `setState` en `useEffect` para sincronizar estado derivado de la URL/props:** la regla `react-hooks/set-state-in-effect` lo prohíbe. Ajusta el estado durante el render comparando con el valor previo (patrón recomendado por React); ver `programs-grid.tsx`.
- **`next.config.ts` lee env al evaluar:** usa `process.env.API_URL ?? process.env.NEXT_PUBLIC_API_URL ?? "http://localhost:4000"` para el destino del proxy.
- **React 19 resetea `<form action={fn}>` tras cada submit:** los inputs **no controlados** vuelven a su estado inicial al completar la action (incluso en fallo). Para conservar un valor tras un intento fallido (p. ej. el correo del login al equivocarse de contraseña) hazlo **controlado** y re-siémbralo desde el estado que devuelve la action — no confíes en `defaultValue`. Ejemplo: `src/app/login/login-form.tsx` (email controlado desde `state.values.email`, re-sincronizado durante el render comparando el valor previo, no en `useEffect`). La server action `authenticate` devuelve el correo **crudo** (antes de validar con Zod) en `LoginState.values.email` para re-hidratar incluso si es inválido.
- **Un Server Component NO puede escribir/borrar cookies en Next 16** — solo los **Route Handlers** y las **Server Actions** pueden. Consecuencia: si un Server Component (p. ej. una lectura admin durante el render) necesita cerrar sesión / limpiar la cookie, no puede llamar a `signOut` directamente; debe **redirigir a un Route Handler** que lo haga. Ver `src/app/api/auth/session-expired/route.ts` (`signOut({ redirect:false })` + `NextResponse.redirect`).
- **"Recordarme" = downgrade de la cookie de sesión en la Server Action.** NextAuth siempre escribe la `authjs.session-token` como cookie **persistente** (`Max-Age = session.maxAge`), marques o no la casilla — así que el flag por sí solo no cambia nada observable. Para que "sin Recordarme" cierre sesión al cerrar el navegador, `src/app/login/actions.ts` llama `signIn("credentials", { ..., redirect: false })` (para recuperar el control tras autenticar) y, si **no** se marcó, **reescribe** la cookie con el mismo valor pero **sin `Max-Age`/`Expires`** (cookie de sesión), vía `cookies()` de `next/headers` (permitido en Server Actions). Maneja la variante `__Secure-` y los fragmentos `.0/.1`. Con `redirect:false`, una credencial inválida **no lanza**: `signIn` devuelve una URL con `?error=` (se detecta para mostrar el error). El TTL real lo sigue gobernando el `exp` del token del backend (1 día vs 30 con `JWT_REMEMBER_EXPIRES_IN`), no el `maxAge`.
- **La sesión de NextAuth puede sobrevivir al `accessToken` del backend → bucle de redirección.** La sesión JWT de Auth.js tiene su propio `maxAge` y es _rolling_ (se renueva en cada request), mientras el `accessToken` del backend tiene TTL fijo (`JWT_EXPIRES_IN`). Si el token caduca/queda inválido (vencido, usuario `isActive=false`, DB reseedeada) pero la sesión sigue "viva", las llamadas admin reciben **401** aunque `auth()` devuelva sesión. Un `redirect("/login")` ingenuo entra en bucle: `login/page.tsx` ve la sesión válida y rebota a `/dashboard` → 401 → `/login` → … (`ERR_TOO_MANY_REDIRECTS`). Solución aplicada (ver `src/auth.config.ts`, `src/lib/api/admin.ts`): (1) `session.maxAge` es la **cota superior** = el token más largo posible (30 días, el de "Recordarme"); el `exp` real del `accessToken` es la fuente de verdad de la duración; (2) los callbacks `authorized` y `session` decodifican el `exp` del `accessToken` (`isBackendTokenExpired`, edge-safe con `atob`, **no** `Buffer`) y lo invalidan proactivamente; (3) los 401 del path admin redirigen a `/api/auth/session-expired` (Route Handler que cierra la sesión muerta y manda a `/login?expired=1`), rompiendo el bucle.
- Recuerda: middleware es `src/proxy.ts` (no `middleware.ts`); ver CLAUDE.md para el contrato de auth.
